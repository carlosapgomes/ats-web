# dashboard-case-audit-report Spec Delta

## ADDED Requirements

### Requirement: Authorized dashboard users shall inspect the reconstructed automatic report for cases handed to a doctor

The dashboard case detail SHALL provide `manager` and `admin` users with a textual reconstruction of the automatic report when the case history proves that the case was handed to a doctor.

#### Scenario: Manager audits a case handed to a doctor
- **GIVEN** a case has a `CASE_READY_FOR_DOCTOR` event
- **AND** its persisted LLM artifacts are available wholly or partially
- **WHEN** a user with active role `manager` opens the dashboard case detail
- **THEN** the page provides a control labelled `Relatório automático apresentado ao médico (reconstruído)`
- **AND** expanding it shows the report context and the seven technical report sections generated by the canonical doctor presenter

#### Scenario: Administrator audits a case handed to a doctor
- **GIVEN** a case has a `CASE_READY_FOR_DOCTOR` event
- **WHEN** a user with active role `admin` opens the dashboard case detail
- **THEN** the reconstructed automatic report is available under the same read-only collapse control

#### Scenario: Case was never handed to a doctor
- **GIVEN** a case does not have a `CASE_READY_FOR_DOCTOR` event
- **WHEN** an authorized dashboard user opens its detail
- **THEN** the page does not claim that an automatic report was presented to a doctor
- **AND** the reconstructed doctor report control is absent

### Requirement: Dashboard and doctor views shall share the canonical report preparation

The system SHALL prepare the doctor report from the same persisted case inputs and prior-denial context for both the doctor decision view and the dashboard audit view.

#### Scenario: Same case is rendered in doctor and dashboard contexts
- **GIVEN** a case has structured data, summary text, suggested action, extracted text and optional recent-denial context
- **WHEN** the doctor view builds the structured report
- **AND** the dashboard builds the textual report
- **THEN** both use the same report-preparation helper
- **AND** both use `DoctorReportPresenter`
- **AND** the dashboard does not duplicate the seven report-building rules

#### Scenario: Recent denial context exists
- **GIVEN** the existing prior-case lookup finds a recent doctor or appointment denial
- **WHEN** the report is reconstructed
- **THEN** the denial type, reason, timestamp and count are prepared through the same mapping used by the doctor view

### Requirement: The reconstructed report shall be collapsed and transparent about its limitation

The dashboard SHALL present the report as collapsed plain text and SHALL disclose that it is reconstructed rather than an immutable historical snapshot.

#### Scenario: Report initial state
- **GIVEN** an authorized dashboard user opens an eligible case detail
- **WHEN** the page first renders
- **THEN** the report body is collapsed
- **AND** the control has `aria-expanded="false"`
- **AND** the control references the report body through a coherent `aria-controls`/collapse target

#### Scenario: Auditor expands report
- **GIVEN** the reconstructed report control is visible
- **WHEN** the auditor activates it
- **THEN** Bootstrap Collapse reveals pre-wrapped readable text
- **AND** no custom JavaScript or new endpoint is required

#### Scenario: Reconstruction disclosure
- **GIVEN** a reconstructed report is available
- **WHEN** the card is rendered
- **THEN** the page explains that the content was reconstructed from stored case artifacts
- **AND** it does not describe the content as an immutable snapshot or byte-exact historical copy

### Requirement: The audit report shall not broaden access or execute generated markup

The feature SHALL preserve existing dashboard authorization and Django output escaping.

#### Scenario: Unauthorized role requests dashboard detail
- **GIVEN** a user has active role `nir`, `doctor` or `scheduler`
- **WHEN** the user requests the dashboard case detail
- **THEN** existing role protection continues to deny dashboard access

#### Scenario: Shared intake template is rendered outside dashboard context
- **GIVEN** `templates/intake/case_detail.html` is rendered without `show_dashboard_nav`
- **WHEN** the case contains LLM artifacts and doctor handoff history
- **THEN** the reconstructed dashboard audit card is absent

#### Scenario: Generated text contains HTML-like content
- **GIVEN** a persisted report input contains HTML or script-like text
- **WHEN** the reconstructed report is rendered
- **THEN** Django escapes the content
- **AND** the template does not apply `safe` to the report text

### Requirement: The change shall remain read-only and non-persistent

The reconstructed report feature SHALL NOT add or modify persisted clinical artifacts or workflow state.

#### Scenario: Auditor views reconstructed report
- **GIVEN** an eligible case exists
- **WHEN** manager/admin opens and expands the report
- **THEN** no `Case` field changes
- **AND** no `CaseEvent` is created
- **AND** no FSM transition occurs
- **AND** no report snapshot is persisted
